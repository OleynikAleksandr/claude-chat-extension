# Инструкции для Claude

## Основные правила работы

### 1. Язык общения
- Всё общение в чате ведётся на русском языке

### 2. Управление задачами
- Любую задачу превращать в микро-задачи через TodoWrite
- Выполнять задачи последовательно, отмечая прогресс

### 2.0. КРИТИЧЕСКИ ВАЖНОЕ ПРАВИЛО
- **НЕ НАЧИНАТЬ НИЧЕГО ДЕЛАТЬ БЕЗ ЯВНОГО ОДОБРЕНИЯ ПОЛЬЗОВАТЕЛЯ**
- **НЕ ПИСАТЬ КОД, НЕ ИСПРАВЛЯТЬ ДОКУМЕНТЫ БЕЗ РАЗРЕШЕНИЯ**
- **ВСЕГДА СНАЧАЛА ОБСУЖДАТЬ ПЛАН ДЕЙСТВИЙ С ПОЛЬЗОВАТЕЛЕМ**
- Перед любыми изменениями кода или документов - спрашивать разрешение
- Предлагать варианты решения, но не реализовывать их без подтверждения

### 3. Начало сессии
При начале любой сессии:
- Найти папку `doc/Sessions/`
- Получить список всех файлов в папке
- Извлечь все номера XXX из названий файлов `отчет_*_XXX.md`
- Найти максимальный номер XXX
- Прочитать файл с максимальным номером
- Если файлов нет - создать все папки и в конце сессии первый отчет
- Сообщить пользователю, что готов к продолжению

### 4. Завершение сессии
При запросе завершить сессию:
- Создать подробный отчёт в папке `doc/Sessions/`
- Формат имени: `отчет_ИмяПроека_XXX.md` (где XXX - номер от 001)
- Структура отчёта:
  - Дата и время сессии (по времени Мадрида, UTC+1/UTC+2)
  - Выполненные задачи
  - Список изменённых файлов
  - Текущее состояние проекта
  - Планы на следующую сессию
  - Возникшие проблемы и их решения
  - Использовать Bash с heredoc для создания файлов с русским текстом:
  ```bash
  cat > /путь/к/файлу.md << 'EOF'
  # Русский текст сохранится корректно
  Содержимое файла с кириллицей
  EOF
  ```

### 5. База знаний
При возникновении проблем, требующих поиска решения:
- Создать запись в папке `doc/kb/`
- **Формат имени**: `kb_ПроблемаИРешение_НомерВерсии.md`
  - Пример: `kb_WriteToolNewFiles_TouchReadWrite_001.md`
  - Пример: `kb_GitCommitErrors_SetUserEmail_001.md`
  - Пример: `kb_NodeModulesPermissions_ChownCommand_001.md`
- **Принцип именования**: 
  - `kb_` - префикс для базы знаний
  - `ПроблемаИРешение` - краткое описание проблемы и ключевого решения
  - `НомерВерсии` - номер от 001, для версионирования решений
- Структура записи:
  - Описание проблемы
  - Попытки решения
  - Финальное решение
  - Выводы и рекомендации

### 6. Создание новых файлов
- **ВСЕГДА** для новых файлов: `touch файл` → `Read файл` → `Write файл`
- Write tool требует предварительного Read даже для несуществующих файлов

### 7. Правила версионирования пакетов
- **КАЖДАЯ ПЕРЕСБОРКА = НОВЫЙ НОМЕР РЕЛИЗА**
- Правильно: 0.4.2 → 0.4.3 → 0.4.4 → 0.4.5...
- НЕ использовать один номер версии для нескольких сборок
- НЕ говорить "заметка на будущее" без записи в документацию
- Всегда обновлять version в package.json перед пересборкой

### 8. Кодировка UTF-8
- **ПРОБЛЕМА:** Write tool не поддерживает корректную UTF-8 кодировку для кириллицы
- **РЕШЕНИЕ:** Использовать Bash с heredoc для создания файлов с русским текстом:
  ```bash
  cat > /путь/к/файлу.md << 'EOF'
  # Русский текст сохранится корректно
  Содержимое файла с кириллицей
  EOF
  ```
- **ВСЕГДА проверять Read** после создания .md файлов с кириллицей
- **Если искажения** - удалить файл и пересоздать через Bash heredoc
- **База знаний:** kb_UTF8EncodingIssues_VSCodeWriteTool_002.md
